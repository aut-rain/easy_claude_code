# Skills：为 AI 助手装上可复用的专业能力

> **Skills（技能）** 是一种把"怎么做某类任务"的程序性知识打包成可复用模块的机制。你可以把它理解为给 AI 助手的**入职手册 / 专业工具包**——助手在需要时自动加载它，从而从通用助手变成某个领域的行家。
>
> 这套机制被 Claude Code、Codex、Cursor、OpenCode 等主流 AI 编程助手广泛支持，并通过开放生态 [Skills.sh](https://www.skills.sh/) 共享。本文不绑定任何单一品牌，讲的是通用的 Skills 概念与实践。

---

## 一、Skills 是什么

一个 **Skill（技能）** 本质上是一个文件夹，至少包含一个 `SKILL.md` 文件，可选地附带脚本、参考资料等资源：

```
my-skill/
├── SKILL.md          # 必需：技能的指令与元信息
├── scripts/          # 可选：配套脚本
│   └── lint.sh
└── references/       # 可选：按需加载的详细资料
    └── style-guide.md
```

它和"写一段很长的提示词"的根本区别在于：

| | 传统提示词 | Skill |
|---|---|---|
| 存在形式 | 每次手动粘贴 | 固化在目录里，随叫随到 |
| 加载时机 | 一开始就占用上下文 | **用到时才加载** |
| 复用 | 复制粘贴 | 一处定义，多处/多人共用 |
| 触发 | 你记得带上 | 助手**自动判断**是否需要 |

换句话说，Skills 把"行业规范、操作流程、工具用法"这类**程序性知识**沉淀下来，让助手不再依赖你每次手动输入冗长指令。

---

## 二、工作机制：渐进式披露（Progressive Disclosure）

Skills 之所以省上下文，靠的是**渐进式披露**：

1. **启动时**：助手只加载每个技能的"目录卡"——即 `name` 和 `description`，并不加载 `SKILL.md` 正文。即便你装了 50 个技能，启动开销也很小。
2. **触发时**：当某次任务匹配到某个技能，助手才把那个 `SKILL.md` 的完整内容注入到当前上下文。
3. **触发方式**有两种：
   - **自动触发**：助手根据 `description` 判断当前任务是否需要某个技能。所以**描述写得好不好，直接决定技能会不会被用上**。
   - **手动触发**：用户输入 `/技能名` 显式调用。

## 三、SKILL.md 的结构

`SKILL.md` 由两部分组成：顶部的 **frontmatter（元信息）** 和下面的**正文（详细指令）**。

### frontmatter 字段

| 字段 | 必需 | 作用 |
|------|------|------|
| `name` | ✅ | 技能名，kebab-case（如 `code-review`） |
| `description` | ✅ | 一句话说明技能做什么、**何时**该用它。助手据此判断是否触发——**这是整个技能最关键的字段** |
| `context` | ❌ | 设为 `fork` 时，技能在隔离的子代理中运行，不污染主上下文（适合重流程） |
| `disable-model-invocation` | ❌ | `true` 表示只能手动调用，不让模型自动触发 |
| `user-invocable` | ❌ | `false` 表示只能由模型自动触发，不能手动 `/` 调用 |
| `license` / `metadata` | ❌ | 许可证与附加元信息 |

### 一个完整示例

```markdown
---
name: code-review
description: 对当前改动做一次结构化代码审查，检查正确性、可读性、性能与安全问题，并给出可执行的改进清单。在用户请求"审查代码 / review / 检查改动"时使用。
---

# 代码审查

按以下步骤审查当前工作区的改动：

1. 先用 `git diff` 看清全部改动范围。
2. 对每个改动文件，从四个维度评估：
   - **正确性**：逻辑是否正确，有无边界遗漏
   - **可读性**：命名、结构、注释
   - **性能**：是否有明显的低效操作
   - **安全**：注入、越权、敏感信息泄露
3. 输出一张表：文件 / 行 / 问题等级 / 建议。
4. 末尾给出"建议优先修复"的三条。

> 详细的安全检查清单见 `references/security-checklist.md`，需要时再读取。
```

注意最后一行：把"偶尔才用的大段资料"放进 `references/`，让助手**按需读取**，而不是一开始就塞进上下文。这正是渐进式披露的精髓。

## 四、创建一个自己的 Skill

### 1. 选好放置位置

| 位置 | 作用域 | 适合 |
|------|--------|------|
| `~/.claude/skills/` | 个人全局 | 你个人常用的、跨项目复用的 |
| `.claude/skills/` | 项目级（随仓库提交） | 团队共享、与项目绑定的 |

（下面以 Claude Code 的约定为例，其他助手路径类似，放在各自的 skills 目录即可。）

### 2. 建目录、写 SKILL.md

比如做一个"提交信息规范"技能：

```
.claude/skills/commit-style/
└── SKILL.md
```

`SKILL.md` 内容：

```markdown
---
name: commit-style
description: 按团队的 Conventional Commits 规范撰写提交信息。在执行 git commit 前使用，确保提交信息符合 type(scope): subject 格式。
---

# 提交信息规范

提交信息格式：`type(scope): subject`

- type 只能是：feat / fix / docs / style / refactor / test / chore
- subject 用中文，不超过 50 字，祈使语气，结尾不加句号
- 若改动涉及破坏性变更，type 后加 `!`，并在正文说明
```

### 3. 验证

启动助手后，让它提交代码，它应自动应用上述规范；或手动输入 `/commit-style` 调用。如果没触发，多半是 `description` 没写清"何时使用"——回去把触发场景描述得更具体。

## 五、发现与安装：Skills.sh

不想自己写？社区里已经有大量现成技能。**[Skills.sh](https://www.skills.sh/)** 是目前最活跃的开放 Skills 目录与包管理器，被称作 **"npm for AI agent skills"**——只不过你安装的不是代码库，而是"专业知识包"。

### 核心命令（通过 npm 调用，无需全局安装）

```bash
# 搜索技能
npx skills find react

# 安装某个技能
npx skills add <skill-name>
```

安装后，技能被放进本地的 skills 目录，**助手会自动识别并按需触发**，无需额外配置。

### Skills.sh 的特点

- **单命令安装**：`npx skills add <skill>`，体验和 `npm install` 一致。
- **跨工具通用**：同一套技能可在 Claude Code、Codex、Cursor、OpenCode 等 60+ 工具中使用，不锁死单一生态。
- **社区驱动**：任何人都能编写并发布技能，目录里有数千个可浏览、可按热度排序的技能。
- **可在线浏览**：访问 [www.skills.sh](https://www.skills.sh/) 按分类、趋势发现技能。

> 💡 选技能时，优先看 `description` 是否清晰、是否有活跃维护——一个好技能的描述应当让你一眼看懂"它何时该被用上"。

## 六、Skills vs MCP vs 插件：别搞混

这三者经常被放在一起，但职责完全不同：

| 机制 | 解决什么问题 | 例子 |
|------|--------------|------|
| **Skills** | "**怎么做**"——程序性知识 / 流程 / 规范 | "怎么做代码审查""怎么写符合规范的提交" |
| **MCP** | "**访问什么**"——连接外部服务 / 数据源 | 连数据库、连 Jira、连 Figma |
| **插件** | "**打包分发**"——把命令、子代理、hooks、MCP 打成一个可安装包 | 一个安装包同时带来若干命令和 hook |

一句话：**写知识用 Skill，连外部用 MCP，打包分发用插件**。它们可以组合——比如一个插件里就捆绑了几个 Skill 和一个 MCP 服务器。

---

## 七、最佳实践

1. **把心思花在 `description` 上**。它决定技能的触发率。务必写清"这个技能做什么"和"什么场景下该用它"，可参考第 3 节示例。
2. **一个技能只做一件事**。粒度小的技能更容易被准确触发，也更好维护。
3. **重流程用 `context: fork`**。如果某个技能要跑很多步、读很多文件，设为 `fork` 让它在隔离上下文里运行，避免撑爆主对话。
4. **大段资料放 `references/`**。正文只放主流程，详细清单、长文档作为引用文件按需读取。
5. **团队技能进仓库**。把 `.claude/skills/` 提交到 Git，全团队 clone 后即用即得，规范自动统一。
6. **记得测触发**。新建技能后，主动构造一个"应该触发它"的场景，确认助手确实加载了它；没触发就去改 `description`。

---

## 小结

Skills 的价值在于把**反复出现的专业知识流程化、可复用化**：写一次，助手就能在合适的时机自动应用。从一个 5 行的提交规范，到一整套代码审查 / 文档生成 / 数据分析流程，都可以沉淀成 Skill。

当你想快速获得能力而不是自己写时，就去 [Skills.sh](https://www.skills.sh/) 找现成的；当你的团队有独特流程时，就把它沉淀成自己的 Skill 放进仓库。长此以往，你的 AI 助手会越来越像一个真正熟悉你团队规矩的同事。
